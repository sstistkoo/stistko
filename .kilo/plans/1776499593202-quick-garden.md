# Plan: Multi-Language Strong's Dictionary Cleanup & Extraction

## Current State Analysis
- **strong_bible_cz.json**: 14,298 entries, format `{"H1": {"definitions_cz": "1) meaning | 2) meaning...", "translated": true}}`
- **stronghebrew.json**: 8,797 Hebrew entries with full lexical data (lemma, translit, morphology, definitions in English)
- **strongsgreek.json**: 5,524 Greek entries (numbered 00001-56243) with full lexical data

## Problems Identified
1. **Multi-meaning entries**: `definitions_cz` contains entire definition strings with `|` separators, not simple word translations
2. **Encoding issues**: Corrupted characters (`�`, `?`, `�`) detected in ~39 entries
3. **Mixed content**: Definitions include grammatical labels (Qal, Piel), explanations, proper names, and editorial notes (`+++`)
4. **No language separation**: All data in one structure, cannot support multiple languages
5. **Inconsistent format**: Some entries are single words, others are multi-line definitions

## Target Structure
```json
{
  "H00001": {
    "lemma_he": "אָב",
    "translit": "ʼâb",
    "meanings": [
      {
        "meaning_en": "father of an individual",
        "translations": {
          "cz": "otec",
          "en": "father",
          "bg": "баща",
          "zh": "父",
          "es": "padre",
          "pl": "ojciec",
          "sk": "otec"
        }
      }
    ],
    "morph": "n-m",
    "pos": "Noun"
  }
}
```
Or simplified:
```json
{
  "H00001": {
    "he": "אָב",
    "en": "father",
    "cz": "otec",
    "bg": "баща",
    "zh": "父",
    "es": "padre",
    "pl": "ojciec",
    "sk": "otec"
  }
}
```

## Implementation Steps

### Phase 1: Data Extraction & Cleaning (Czech)
1. **Parse `definitions_cz` entries**:
   - Split by `|` to separate meanings
   - Extract the first lemma/translation from each meaning (text before first `)` or `=`)
   - For proper names (e.g., `"Abagtha = \"Bohem dáno\""`), use the name itself
   - Strip grammatical labels: `(Qal)`, `(Piel)`, `(Hiphil)`, `n m`, `adj`, etc.
   - Remove `+++` markers and editorial notes
   - Fix encoding artifacts (replace `?`, `�` with proper characters or omit)

2. **Create mapping script** (Python/Node):
   - Read `strong_bible_cz.json`
   - For each entry, extract the primary Czech word (first meaning before `|`, cleaned)
   - Output: `strong_translations_cz.json` with `{"H1": {"cz": "otec"}}`

3. **Validate against Hebrew/Greek source**:
   - Cross-check that all H/G numbers in Czech file exist in Hebrew/Greek lexicons
   - Report missing numbers

### Phase 2: Add Source Language Data
4. **Extract Hebrew lemmas** from `stronghebrew.json`:
   - For each H entry, grab: `lemma` (hebrew), `xlit` (transliteration), `POS`, `morph`
   - Add to translation file: `{"H1": {"he": "אָב", "translit": "ʼâb", "pos": "n-m"}}`

5. **Extract Greek lemmas** from `strongsgreek.json`:
   - For each G entry, grab: `greek.unicode`, `pronunciation`, `definition` (short)
   - Add to translation file: `{"G00001": {"gr": "Α", "translit": "A"}}`

### Phase 3: Multi-Language Expansion
6. **Add existing languages** from current data:
   - English (en) – from Hebrew/Greek `definitions` field (first meaning)
   - Bulgarian (bg), Chinese (zh), Spanish (es) – **NOT PRESENT** in current files, need external sources

7. **Integrate Polish (pl) and Slovak (sk)**:
   - **Option A**: Use public domain Bible translations (Biblia Tysiąclecia, Slovenská katolícká)
   - **Option B**: Use existing Strong's dictionaries (search for `strongs_polish.json`, `strongs_slovak.json`)
   - If unavailable, manually create minimal entries for most common words (top 1000-2000)

8. **Add missing major languages** that appear in Bible translation context:
   - Russian (ru) – easy to source
   - German (de) – Luther Bible public domain

### Phase 4: File Organization
9. **Chunk the data** (optional but recommended):
   - Split into 10-20k entry chunks: `strong_translations_H00001-H20000.json`, etc.
   - Or use directory structure: `/H/H00001-H10000/`, `/H/H10001-H20000/`, `/G/`
   - Create `strong_translations_index.json` mapping numbers to file locations

10. **Create database dump** (optional):
    - Export to SQLite with columns: `strong_num`, `language`, `translation`, `meaning_index`
    - Enables efficient queries: `SELECT * WHERE strong_num='H1' AND language='cz'`

### Phase 5: Quality Assurance
11. **Validation script** to check:
    - No duplicate Strong numbers within a file
    - All required fields present (he/gr lemma + at least one translation)
    - No empty translation strings
    - Character encoding is UTF-8 clean (no � replacement chars)
    - Cross-validate: same Strong number has same base word across languages (no contradictions)

12. **Test integration**: Load sample verses (like Gen 1:1-2) and verify all Strong-coded words resolve to translations

## Estimated Scope
- **Total Strong numbers**: ~14,300 entries to process
- **Work per entry**: 1-5 seconds (parsing, cleaning, mapping)
- **Time**: ~1-2 hours for full extraction + cleanup
- **Chunk size**: Recommend 5,000 entries per file (28 files for H, 6 for G)

## Deliverables
1. `strong_translations_core.json` – clean Czech extracted translations (all 14,298 entries)
2. `strong_hebrew_lemmas.json` – Hebrew lexical data (8,797 entries)
3. `strong_greek_lemmas.json` – Greek lexical data (5,524 entries)
4. `strong_translations_multi.json` – combined multi-language structure
5. `/chunks/` directory with split JSON files
6. `validate.py` / `validate.ps1` – data validation script
7. `README.md` – structure documentation

## Risks & Mitigations
- **Missing Polish/Slovak**: If no quality source found, skip initially, add placeholder, or use machine translation with disclaimer
- **Encoding corruption**: Some entries may need manual fix (list produced by validation)
- **Ambiguous mappings**: Multi-meaning words – keep as array of objects rather than single string
- **Performance**: Large files (>10MB) – chunking solves this
